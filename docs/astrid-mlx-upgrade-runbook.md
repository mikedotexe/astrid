## Upgrade Shape

Use a parallel candidate venv first. Do not mutate live `8090` until the
candidate has passed the gates.

Example for a future MLX release:

```bash
cd /Users/v/other/neural-triple-reservoir
python3 -m venv .venv-mlxNEXT
.venv-mlxNEXT/bin/python -m pip install --upgrade pip
.venv-mlxNEXT/bin/python -m pip install mlx==NEXT mlx-metal==NEXT mlx-lm==PINNED_OR_NEXT
.venv-mlxNEXT/bin/python -m pip show mlx mlx-metal mlx-lm
```

If the candidate needs the rest of the live dependency set, seed it from the
current venv, then reinstall the candidate MLX wheels last.

```bash
  --reservoir-root /Users/v/other/neural-triple-reservoir \
```

artifacts, malformed `NEXT:` findings, deprecated runtime wording generated by
Astrid, fallback incidents, or major latency regression.

```bash

  --python /Users/v/other/neural-triple-reservoir/.venv-mlxNEXT/bin/python \
```

the model generation log.


accepts a known non-MLX regression.

Promotion means upgrading the live venv, then restarting only the coupled Astrid
LaunchAgent:

```bash

  mlx==NEXT mlx-metal==NEXT mlx-lm==PINNED_OR_NEXT

```

After promotion, run a calm live-turn watch first. Escalate to a full soak only
if the watch shows suspicious behavior.
